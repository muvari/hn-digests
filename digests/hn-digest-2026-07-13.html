1 Security

GhostLock, a stack-UAF that has existed in all Linux distributions for 15 years

397 points · 196 comments · nebusec.ai

Nebula Security details GhostLock (CVE-2026-43499), a use-after-free bug in the Linux kernel's rtmutex/futex requeue-PI code that has existed in every distribution since Linux 2.6.39 (2011) and was only patched this April. The helper function remove_waiter() assumed it was always cleaning up the currently running thread's own waiter object, but the proxy-lock path used by FUTEX_CMP_REQUEUE_PI lets it clear the wrong task's state, leaving a dangling pointer into a thread's own kernel stack. Any unprivileged local user can trigger it with ordinary futex syscalls, and the researchers turned it into a 97%-reliable privilege escalation and container escape, earning $92,337 from Google's kernelCTF.

What the comments said

  • Several commenters were confused by the jargon-heavy title and structure, noting the post buries what "stack-UAF" and the actual impact are until deep into the write-up.
  • There was debate over whether "use-after-free" is a term LLMs invented versus genuine long-standing infosec vocabulary; most experienced commenters confirmed it's a decades-old, very standard bug-class name.
  • Readers worried the "LPE" (local privilege escalation) framing undersells the risk, since it enables container escape and is paired with an unrelated Firefox/IonMonkey type-confusion bug (CVE-2026-10702) that a commenter demoed turning a malicious webpage into root access on Android.
  • Several users reported testing the linked demo exploit on Android devices, with mixed results (boot loops, forced shutdowns), and discussed how sensitive the exploit is to per-build stack layout and compiler behavior.

Are you telling me a readonly property is wrecking my performance?

62 points · 37 comments · shub.club

While debugging a slowdown in Letta Desktop, the author traces it to a common anti-pattern: reading `el.scrollHeight` to auto-scroll a chat UI on every new message. Despite being a nominally "readonly" property, scrollHeight forces the browser to synchronously recompute layout on read (as shown in the Chromium source), so calling it repeatedly during a stream of incoming messages triggers expensive layout thrashing; the fix was simply setting scrollTop to an oversized constant instead of computing an exact scrollHeight value.

What the comments said

  • Several commenters generalized the lesson: any layout-related DOM property (scrollHeight, offsetHeight, getComputedStyle, etc.) can force a synchronous layout recalculation on read, and interleaving style writes with layout reads in a loop is one of the most common web performance bombs.
  • One commenter cited virtual DOM's popularity as partly a response to exactly this problem, since it batches DOM writes and reads for you rather than requiring manual care.
  • A tangential debate arose about whether JavaScript's `let`/`const` (versus `var`) carry a measurable runtime performance penalty due to the temporal-dead-zone check, with mixed opinions on how significant it is in practice.
  • A broader architectural point: `readonly`/encapsulation is meant to signal safety and immutability to callers, but that guarantee says nothing about the cost of computing the value underneath, and developers often wrongly assume the two go together.
