2 AI & Dev Tools

Claude Code sends 33k tokens before reading the prompt; OpenCode sends 7k

626 points · 334 comments · systima.ai

A benchmark from Systima instrumented Claude Code and OpenCode with a logging proxy to compare exactly what each harness sends to the model. It found Claude Code's first request carries roughly 33,000 tokens of system prompt, tool schemas, and scaffolding versus OpenCode's ~7,000, and that Claude Code rewrites cache tokens far more often, making it less cache-efficient run to run. Adding a large CLAUDE.md/AGENTS.md file and a few MCP servers pushes a real working setup to 75,000–85,000 tokens before a user even types anything, and fanning work out to subagents multiplies the overhead sharply—though Claude Code can still win on total task cost by batching tool calls into fewer requests.

What the comments said

  • Multiple commenters were skeptical of the methodology, questioning the choice of an old pinned model and why the authors' own custom LLM gateway seemed to interfere with parts of the test; the author replied defending the model choice as a cost decision and clarified the gateway's role.
  • Several agreed the finding tracks their own experience that agent harnesses are becoming aggressive tool-callers even for trivial prompts like "hey" or "commit," calling it "tokenflation."
  • Commenters debated whether Anthropic is economically incentivized toward higher token usage, and whether that actually correlates with better agent performance or is just padding.
  • A few pointed to minimalist alternative agents with much smaller system prompts as a counterexample, though others noted such minimal agents ship without many practical tools.
  • One thread flagged a competing context-pruning tool as opaque and likely to break prompt caching.
3 AI Infrastructure

Mesh LLM: distributed AI computing on iroh

340 points · 90 comments · iroh.computer

Mesh LLM is an open-source system that pools GPUs across many machines into a single OpenAI-compatible API, built on the iroh peer-to-peer networking library. Each node exposes a public-key identity over encrypted, NAT-traversing QUIC, and requests can run locally, route to a peer that already has the model loaded, or split a large model layer-by-layer across a pipeline of machines ("Skippy" mode) so no single box needs to hold the whole thing. There's no central server; a gossip layer built atop iroh handles routing, peer discovery, and trust.

What the comments said

  • Commenters debated real-world throughput: some expected network splitting to be far too slow for interactive use, while a project contributor and others cited concrete numbers (10–16 tokens/sec on a 235B MoE model split across nodes) as surprisingly usable.
  • Several noted that per-token network traffic for split inference is small (just activation vectors), so latency rather than bandwidth is the binding constraint.
  • Questions arose about encryption and payload privacy between nodes, with the answer that iroh's key-based dialing provides encryption by default.
  • One commenter contrasted it with existing offerings like cocompute.ai, arguing those solve a different problem (exposing local models to the internet) rather than true model-splitting.
  • A project contributor showed up in the thread to answer questions directly, including about fault tolerance if a node drops mid-inference.

20 Security

Since Chromium 148, Math.tanh is now fingerprintable to link underlying OS

407 points · 196 comments · scrapfly.dev

Scrapfly documents a newly fingerprintable browser signal: since Chrome 148, V8 stopped computing Math.tanh with its own bundled, OS-independent math routine and switched to calling the host operating system's native math library (glibc on Linux, libsystem_m on macOS, UCRT on Windows)—libraries that disagree on the last bit or two of precision for many inputs. Because that disagreement is deterministic and OS-specific, a single well-chosen Math.tanh() call (plus related CSS trig functions, which have leaked this way even longer) can betray a browser's true underlying operating system, undermining any anti-bot setup that spoofs its User-Agent OS while actually running on a different one—and Apple Silicon's two internal math libraries even disagree with each other depending which code path a given API uses.

What the comments said

  • The clearest practical takeaway commenters drew out: since most people don't spoof their OS in the User-Agent, this mainly matters for bot/scraping operators impersonating a specific OS—several noted that's precisely the audience the publishing company (a scraping-infrastructure vendor) serves.
  • Many commenters criticized the post's obviously LLM-generated prose style, some appreciating the disclosed-by-design transparency about AI-assisted writing while others said it undermined trust in the underlying technical claims.
  • A tangent discussed correctly-rounded transcendental math functions as a mostly-solved research problem, with speculation about why JS engines don't just universally keep using the old OS-independent fdlibm implementation.
  • Some pointed out the finding is really more useful for narrowing browser/Chromium version ranges than definitively pinning OS, since there are many other, more reliable version-fingerprinting signals already in wide use.
