Sleep regularity is a stronger predictor of mortality risk than sleep duration (2023)

693 points · 368 comments · academic.oup.com

The publisher blocked automated access to this 2023 paper from the journal SLEEP, but per its title and the ensuing discussion, it argues that how consistent a person's daily sleep timing is predicts all-cause mortality better than how many hours they sleep. Commenters quoting the text describe a large cohort (roughly 100,000 of the ~500,000-participant UK Biobank), with results "adjusted for age, sex, ethnicity, and sociodemographic, lifestyle, and health factors," using accelerometer-derived sleep estimates rather than direct sleep-stage measurement. What the comments said

  • Many lifelong night owls and irregular sleepers reacted with dark humor or worry, sharing decades-long histories of erratic sleep schedules that "should" have killed them by the study's logic.
  • A large thread debated confounding: commenters suggested people with regular sleep may simply have less stressful jobs, healthier general habits, or occupations (e.g., frequent flying, shift work, chemical exposure) that independently affect mortality risk.
  • A prolonged, somewhat contested "correlation vs. causation" car-analogy exchange broke out, with several replies arguing the analogy itself was flawed.
  • Some commenters described successfully "fixing" irregular sleep through disciplined habits (consistent bedtimes, no screens before bed) over weeks to months, while others described biological/genetic sleep-phase issues (delayed sleep phase, ADHD-linked drift) that felt largely outside their control.
  • A few pushed back on methodology, noting the UK-only, roughly 100k-person sample and the use of accelerometer proxies rather than verified sleep states as limitations.

10 Security

TS-2026-009: Insecure argument handling in Tailscale SSH permitted root access

213 points · 145 comments · tailscale.com

Tailscale disclosed that its SSH feature previously accepted usernames beginning with a dash, and on Linux passed them unsanitized as an argument to getent(1). Connecting with the username "-i" was interpreted as the flag --no-idn, causing getent to dump the full passwd file starting with root and letting Tailscale open an interactive root session — bypassing ACL rules meant to restrict users to autogroup:nonroot. The bug is fixed in Tailscale 1.98.9 by rejecting usernames with a leading dash.

What the comments said

  • Commenters recognized this as a decades-old class of argument-injection bug, with wry comparisons to similar flaws going back to AIX in the 1990s.
  • Several were surprised the SSH code path had the ability to grant direct root access at all, questioning why that capability existed by default.
  • Many argued the fix itself is inadequate, merely blocking leading dashes rather than using "--" to separate arguments or avoiding shelling out to getent entirely in favor of getpwnam(3) or parsing unmodified getent passwd output.
  • A side discussion covered why Tailscale self-publishes bulletins with its own numbering instead of requesting a CVE, with some noting this gives vendors more control over disclosure timing, citing curl's move to become its own CNA as precedent.
  • Some reassured others that plain OpenSSH access over Tailscale (not the Tailscale SSH wrapper) is unaffected, and several heavy Tailscale users said they've avoided the SSH feature altogether, preferring OpenSSH's longer track record.

17 Databases

SQLite should have (Rust-style) editions

268 points · 112 comments · mort.coffee

The author argues that SQLite's default configuration is dangerous for real applications: foreign key constraints are off by default, and because SQLite recycles ROWIDs after deletion, a forgotten cascade delete can leave a dangling reference that silently resolves to the wrong (newly inserted) row instead of throwing an error. A second flaw covered is that SQLite's flexible type affinity lets columns silently accept the wrong data type unless the table is explicitly marked STRICT. The proposed fix is Rust-style "editions": an opt-in pragma that switches a database over to a new, safer set of defaults (foreign keys on, strict typing, etc.) without breaking existing code that doesn't opt in.

What the comments said

  • The author showed up in the thread; commenters flagged a small typo in the post and suggested cross-posting the proposal to the official SQLite forum, where the core dev team is known to respond directly to suggestions.
  • A major concern was interoperability: since SQLite database files often get opened by different SQLite versions across machines (an app's bundled version vs. the system sqlite3 CLI), opt-in "editions" could break older tools reading newer files unless the versioning is handled carefully.
  • Many agreed the proposed defaults (foreign keys, strict typing, WAL mode, busy_timeout) already match what serious SQLite users configure by hand every time, making a codified "edition" pragma feel like a low-risk win — Postfix's gradually-evolving "compatibility_level" setting was cited as prior art for this pattern.
  • A side-thread found SQLite's ROWID reuse itself alarming, with one commenter calling it a potential security issue since data could end up misattributed to the wrong user if foreign keys aren't enforced.
  • Some pushed back on parts of the piece, defending SQLITE_BUSY's current behavior as intentionally leaving retry logic to the caller, and noting that "SQLite locks on write" is a common misunderstanding — all databases lock on write, it's really about lock granularity.
18 Security

I tricked Claude into leaking your deepest, darkest secrets

639 points · 289 comments · ayush.digital

Ayush Paul describes finding and exploiting a data-exfiltration path in claude.ai's memory system, which stores a high-fidelity, information-dense profile of each user (built from a daily conversation summary plus a searchable history). Direct requests to make Claude's web_fetch tool visit an attacker-controlled URL with private data embedded were blocked, but Paul discovered web_fetch is allowed to follow hyperlinks it has already seen on a previously fetched page. By building a site with pages for every letter of the alphabet that branch into further letter-pages, he got Claude to "type out" a user's name, employer, hometown, and security-question answers one character at a time by clicking through the letter-tree, silently logging the stolen data to his own server via the URL paths requested. He disclosed the bug through Anthropic's HackerOne program; Anthropic confirmed they already knew about it internally, paid no bounty, and later mitigated it by disabling web_fetch's ability to follow links on external pages outside of search results or user-provided URLs.

What the comments said

  • There was strong frustration that Anthropic paid no bounty despite confirming the vulnerability via HackerOne, with commenters debating whether "we already knew internally" is a legitimate reason to withhold payment or just a way to dodge payouts — Anthropic has since patched the issue by disabling web_fetch's link-following on external pages.
  • Some said they keep Claude's memory feature turned off entirely, either over security concerns or because they find the recalled details more annoying than useful in everyday conversations.
  • A proposed mitigation was routing memory-search results through a subagent that doesn't have full access to raw memories, so it could flag a suspicious attacker-controlled "authorization" request rather than act on it directly.
